Проблема, которую закрывает проект

Агент умеет писать код. Команде нужен контур исполнения.

gigalphex добавляет поверх GigaCode дисциплину рабочего процесса: что именно делаем, когда считаем готовым, где смотреть след выполнения и как не потерять контроль над git-состоянием.

  • Большая задача дробится на независимые markdown-секции.
  • Каждая итерация привязана к одному выбранному разделу плана.
  • Успех требует проверки, обновленного чеклиста и коммита.
  • Ревью и финализация выполняются отдельными фазами, а не “на глаз”.

before-after.diff structured loop

Без раннера

Один длинный prompt, неочевидная граница готовности, трудно понять, что уже проверено и где агент остановился.

С gigalphex

План является контрактом, progress log является следом, а git-коммиты фиксируют атомарные результаты.

Риск

Агент может уйти в соседнюю задачу, оставить грязное дерево или принять ложное ревью-замечание.

Контроль

Выбор ровно одной задачи, dirty guard, retry guard, structured findings и независимая проверка synthesis-фазой.

Основной сценарий

Пайплайн выглядит как маленькая CI-система для агентской разработки

Каждая фаза имеет собственный prompt, модель, лимиты, логику retry и ожидания по результату.

1

Plan

Создать или прочитать markdown-план в `docs/plans/`.

2

Task loop

Выбрать первый незавершенный `Task` или `Задача`.

3

Validate + commit

Выполнить код, тесты, чекбоксы и атомарный коммит.

4

Review fan-out

Запустить пять read-only review-агентов параллельно.

5

Synthesis + finalize

Проверить claims, исправить подтвержденное, закрыть прогон.

1:45 - 2:45

План как контракт

Главный артефакт - markdown-план с чекбоксами

Runner понимает английские и русские заголовки: `Task`, `Iteration`, `Задача`, `Итерация`. Это важно для локальных команд, где планы пишутся на русском, но исполняются тем же механизмом.

  • Выполняется только первый незакрытый task-раздел.
  • Поздние задачи нельзя отмечать или менять раньше времени.
  • Чекбокс `[x]` разрешен только после реализации и проверки.
  • При retry план возвращается к снимку начала итерации.

Task execution

Одна итерация - один контролируемый агентский запуск

gigalphex не пытается заменить интеллект GigaCode. Он задает жесткий контракт вокруг запуска: scope, validation, commit, отсутствие новых незакоммиченных изменений.

S

Scope lock

В prompt передается выбранный task-раздел и запрет работать с поздними задачами.

V

Validation

Агент обязан выполнить релевантные проверки перед закрытием чекбоксов.

G

Git commit

Готовая итерация должна оставить коммит и чистое рабочее дерево.

R

Retry guard

Транзиентные ошибки и rate limits обрабатываются отдельно от настоящего failure.

L

Progress log

Лог пишет lifecycle событий без утечки содержимого prompt.

T

Statistics

JSON рядом с логом хранит время, модели, токены и статус прогона.

Quality gate

Ревью - это параллельная команда специалистов

По умолчанию запускаются пять read-only review-агентов. Они возвращают машинно-валидируемые findings, а synthesis-фаза независимо проверяет claims и только потом исправляет код.

  • `quality` ищет баги, регрессии и безопасность.
  • `implementation` проверяет соответствие требованиям и архитектуре.
  • `testing` фокусируется на тестах и validation gaps.
  • `simplification` ловит лишнюю сложность.
  • `documentation` смотрит docs, comments и user-facing изменения.

Operational controls

Проект заботится о рабочем дереве, ветках и корпоративных правилах

Это особенно важно для реальных задач, где запуск агента должен уважать текущую ветку, Jira-префиксы, worktree-изоляцию и политику dirty working tree.

Git hygiene

Создание или переключение ветки из имени плана, запрет dirty tree по умолчанию, перенос завершенных планов в `completed/`.

Worktree mode

`--worktree` запускает выполнение в `.gigalphex/worktrees/<branch>`, не дергая текущий checkout.

Jira policy

`--jira-task PROJ-123` навязывает `feature/PROJ-123-...` и префикс commit subject.

Timeouts and retries

Session timeout, idle timeout, retry patterns, rate limit patterns и ожидание перед повтором.

Настраиваемость

Все важные границы вынесены в конфиг, env и CLI flags

gigalphex не требует “магического” SDK. Его контракт - командная строка GigaCode и набор настраиваемых prompt templates.

  • Приоритет: embedded defaults, global config, project config, explicit config, `GIGALPHEX_*`, CLI.
  • Для фаз можно выбрать разные модели: plan, task, review, finalize.
  • Prompt overrides лежат глобально или локально в `.gigalphex/prompts/`.
  • `--gigacode-arg` прокидывает существующие флаги GigaCode в каждый запуск.

Как жить рядом с существующей инфраструктурой

gigalphex не заменяет настроенный GigaCode-стек. Он становится внешним оркестратором.

Смысл интеграции: оставить внутри GigaCode уже настроенные доступы, sandbox, tools, subagents и custom skills, а наружу добавить воспроизводимый жизненный цикл задачи.

C

CLI boundary

Вызов идет через `gigacode` в `PATH`. Auth, workspace, approvals и sandbox остаются на стороне GigaCode.

A

Subagents inside

Если в GigaCode уже настроены внутренние subagents, они продолжают работать внутри task/review/finalize-сессий.

S

Custom skills

Собственные skills остаются в каталоге GigaCode. `--skill-dir` позволяет явно указать нужное место.

P

Planning skill

Встроенный skill нужен только для интерактивного создания планов и ставится как обычный GigaCode skill.

M

Models per phase

План, задачи, read-only review, synthesis и finalize можно вести разными моделями.

I

Existing flags

`--gigacode-arg` прокидывает корпоративные настройки, например include-directories или ограничения tools.

8:00 - 9:20

Итог

Главная идея: агентская разработка становится проверяемым процессом

gigalphex дает простой продуктовый слой вокруг GigaCode: план как контракт, progress log как наблюдаемость, Git как журнал решений, review-агенты как quality gate.

  • Для разработчика: меньше ручной координации и больше прозрачности.
  • Для команды: повторяемый сценарий запуска на реальных задачах.
  • Для инфраструктуры: совместимость через существующий GigaCode CLI и skills.
